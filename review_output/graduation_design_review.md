# 毕业设计符合性检查报告

## 1. 项目总体判断

当前项目围绕“知识融入大语言模型的智能用户故事生成系统”提供了较完整的前后端代码、SQLite 知识库、README、论文和答辩 PPT，整体不是只有页面壳子，能够看出“自然语言需求输入、知识增强、用户故事生成、任务拆解、依赖图展示、评价、代码骨架生成”等核心设计。但本地运行验证存在明显阻碍：后端当前环境缺少 FastAPI，`backend/requirements.txt` 混入启动命令；前端打包因 `vue-cli-service` 缺少执行权限失败；云部署链接未能成功访问。论文和 PPT 的总体方向与系统一致，但多处表述明显强于本地代码实现，尤其是多轮推理、语义角色标注、规则引擎、知识图谱、批注修改、局部重推理、CSV/JSON 导出、时间轴视图、完整测试结果等内容，本地材料支撑不足。综合判断：项目基本框架已具备，但答辩前必须补强运行、部署、测试证据和文档一致性。

## 2. 已识别的主要材料

- 代码目录：`backend/`，包含 `main.py`、`llm_module.py`、`kb_module.py`、`evaluation_module.py`、`code_module.py`、`database.py`、`knowledge_models.py`、`seed_common_knowledge.py`、`user_story_kb.db`。
- 前端目录：`frontend/`，Vue2 项目，包含 `src/APP.vue`、`StoryList.vue`、`TaskList.vue`、`TaskGraph.vue`、`package.json`、`package-lock.json`、`node_modules/`。
- 论文文件：`paper/知识融入大语言模型的智能用户故事生成系统的设计与实现.docx`，可抽取到完整章节结构，文档包内包含约 30 张图片和 27 个表格。
- PPT 文件：`ppt/知识融入大语言模型的智能用户故事生成系统.pptx`，共 20 页。
- README/说明文档：根目录 `README.md`，覆盖功能介绍、运行说明、部署说明、接口说明和 FAQ；未找到单独的用户手册、测试说明或部署实测记录文件。
- 数据库/知识库材料：`backend/user_story_kb.db` 中存在 `knowledge_module` 表，本地查询到 38 条模块记录；`backend/data/knowledge.json` 也存在一份 JSON 知识库样例。
- 部署平台链接：已提供 EdgeOne 链接，但本次检查访问失败，原因见第 8 节。
- GitHub 链接：提供了仓库链接，HTTP 层面可重定向到仓库页面；本次结论主要基于本地材料。

## 3. 项目主要优点

- 代码结构基本完整，前端、后端、数据库模型、知识库、模型调用、评价和代码生成模块都有独立文件。
- 后端接口覆盖面较完整，包括健康检查、知识模块管理、知识增强预览、用户故事生成、批量生成、生成并评价、代码片段反推需求、代码骨架生成。
- 前端不是简单静态页面，包含输入、结果展示、故事卡片、任务筛选、ECharts 依赖图、代码生成结果展示、Markdown 导出等交互。
- 论文围绕本项目展开，章节包含背景、需求分析、系统设计、实现、测试和总结，并引用了本地代码文件和接口设计。
- README 说明较完整，能够让检查者理解系统目标、技术栈、运行步骤、接口和常见问题。

## 4. 主要问题

1. 本地运行链路未闭合。当前 Python 环境无法导入 FastAPI；`backend/requirements.txt` 第 6 行写入了 `uvicorn main:app --reload`，这不是合法依赖；前端打包因 `node_modules/.bin/vue-cli-service` 缺少执行权限失败。
2. 云部署未验证成功。提供的 EdgeOne 链接在访问时出现 TLS 连接异常，无法确认线上页面是否可打开，更无法确认核心生成流程是否可演示。
3. 前端接口地址硬编码为 `http://127.0.0.1:8000`，如果只部署静态前端到云平台，用户浏览器会请求本机 8000 端口，线上生成流程大概率不可用，除非另有未提交的反向代理或后端部署配置。
4. 论文/PPT 存在多处实现夸大。多轮推理、语义角色标注、规则引擎、向量相似度匹配、DAG 规则校验、跨故事资源冲突检测、时间轴视图、批注修改、局部重推理、CSV/JSON 导出等功能，本地代码未看到对应实现。
5. 测试支撑偏弱。未发现自动化测试目录、测试脚本、接口测试记录、性能测试日志或可复现实验数据；论文测试表中仍有“待填写”，性能结果和 100% 成功率缺少本地证据。
6. 文档存在不一致。README 写 30+ 知识模块，PPT 写 42 个模块，本地 SQLite 查询为 38 条；论文表格提到 `micro_story` 和 `generate_micro_story`，但当前代码没有该字段/函数，且提示词要求不要生成微型故事。
7. 技术栈表述不一致。前端代码使用 Element UI，论文/PPT 的测试环境处出现 Element Plus 表述。
8. README 部署说明和代码不完全匹配。README 的 Nginx 示例代理 `/api`，但前端实际请求绝对地址 `http://127.0.0.1:8000`，需要统一。

## 5. 代码与系统实现检查结果

### 已有基础

- 后端采用 FastAPI 组织，`main.py` 中有统一响应结构、请求模型和核心路由。
- `llm_module.py` 调用 DeepSeek API，要求模型输出严格 JSON，并对结构化需求、用户故事、任务类型和依赖字段做归一化。
- `kb_module.py` 从 SQLite 知识库读取模块，通过模块名和别名做关键词命中，生成知识增强上下文。
- `evaluation_module.py` 提供结构化完整度、用户故事质量、任务质量、依赖质量等评分。
- `code_module.py` 可根据选中用户故事和任务生成前端、后端、数据库代码骨架。
- 前端 Vue 页面实现了双输入、示例填充、接口调用、故事列表、任务表、依赖图、评分展示、代码生成展示和 Markdown 导出。
- SQLite 数据库中存在知识模块表和 38 条模块数据，说明不是只有空库。

### 存在问题

- 后端依赖不可直接安装：`requirements.txt` 中混入启动命令。
- 当前检查环境无法导入 `fastapi`，后端未能实际启动验证。
- 前端构建失败，原因是 `vue-cli-service` 文件权限为 `rw-r--r--`，无法执行。
- 前端请求地址硬编码本地后端，缺少 `.env` 或构建时配置，部署适配不足。
- 没有看到自动化测试、接口测试脚本、性能测试脚本或日志输出目录。
- 代码中的知识匹配主要是关键词/别名包含匹配，没有论文中描述的向量相似度、规则引擎、多轮验证反馈。
- 系统结果主要依赖外部 DeepSeek API，未提供离线兜底样例、演示数据缓存或答辩演示脚本。

### 建议补充

- 修正 `backend/requirements.txt`，移除启动命令，补齐确定版本或至少保证 `pip install -r backend/requirements.txt` 可成功。
- 明确后端启动目录和命令，例如从 `backend/` 执行 `uvicorn main:app --reload --host 127.0.0.1 --port 8000`。
- 修复前端构建权限或重新安装依赖，保证 `npm run build` 可执行。
- 将前端 API 地址改为环境变量或相对路径，部署时通过反向代理连接后端。
- 增加最小测试材料：接口测试脚本、典型输入输出 JSON、性能测试记录、截图目录。
- 对论文中确实未实现的高级功能，要么补实现，要么降级为“未来工作/设计思路”，不要作为已实现成果。

## 6. 论文检查结果

### 与系统一致的部分

- 论文主题、系统名称、技术路线与本地代码总体一致。
- 论文中提到 FastAPI、Vue、SQLite、DeepSeek、知识库、用户故事、任务拆解、依赖图、评价模块、代码生成等，在代码中都有一定对应。
- 第 4、5 章引用了 `main.py`、`kb_module.py`、`llm_module.py`、`TaskGraph.vue`、`code_module.py` 等文件，能看出论文不是完全脱离代码写作。
- 第 6 章包含接口测试、知识增强测试、功能测试、异常处理、性能测试和验收测试框架。

### 可疑或偏空泛的部分

- 论文大量描述“分句、命名实体识别、语义角色标注、五元组结构化、置信度阈值、多轮迭代推理、规则引擎、向量匹配、DAG 校验、知识图谱实时更新”等机制，本地代码没有看到对应的独立算法或实现。
- 论文写到用户可批注、修改、局部重推理、术语建议弹窗、CSV/JSON 导出、时间轴视图、执行时序视图等，当前前端代码只看到 Markdown 导出、故事选择、任务筛选和依赖图展示。
- 论文表 4.3 写 `generate_micro_story`，表 4.4 和测试表写 `micro_story`，但当前 `llm_module.py` 没有该函数，提示词还明确要求不要生成微型故事。
- 论文测试表中部分证明材料为“待填写”，性能结果缺少脚本、日志或截图支撑。
- 论文中部分图位在文本抽取中显示为空占位，虽然 docx 内实际含图片，但仍需人工确认每张图是否清晰、是否与图题和正文描述一致。
- 测试环境写 Element Plus，但本地依赖是 Element UI。

### 建议如何修改

- 将论文“已实现”范围收敛到本地代码真实具备的能力：关键词/别名知识匹配、Prompt 约束、结构化 JSON 输出、任务依赖字段、ECharts 图展示、评分和 Markdown 导出。
- 对未实现的高级能力统一改为“设计展望”或“后续优化”，避免答辩时被追问无法演示。
- 删除或改写 `micro_story` 相关内容，除非补齐代码和界面展示。
- 将知识库数量统一为实际值，或重新初始化并提供一致统计。
- 补齐第 6 章测试证据：真实请求、真实响应、截图、性能记录、验收表“是否通过”。
- 增加一组“知识增强前后对比”实验，用相同需求对比未命中知识库和命中知识库的输出差异，突出选题特色。

## 8. 部署平台检查结果

- 是否访问成功：未成功验证。浏览访问失败；使用 `curl -I -L` 和 `curl -k --http1.1 -L` 访问 EdgeOne 链接均在 TLS 连接阶段报 `SSL routines::unexpected eof while reading`。
- 能看到哪些功能：当前未能看到页面内容，线上功能无法判断。
- 是否能支撑答辩展示：当前材料不足以判断。结合代码看，前端硬编码请求 `http://127.0.0.1:8000`，如果云平台只部署静态前端而没有可访问后端，答辩现场无法完成生成流程。
- 还缺什么：需要提供可访问的前端地址、后端地址或统一反向代理；需要在部署环境验证 `/health`、`/preview_context`、`/generate_and_evaluate`、`/generate_code` 等主流程；需要准备离线演示数据以防 API Key 或网络异常。

## 9. 是否满足毕业设计基本要求

判断等级：B. 基本框架已具备，但还有较明显短板，需要补强。

理由：该项目有真实代码量和基本工程结构，论文、PPT、README 与系统主题总体一致，能够支撑一个“知识增强的用户故事生成工具”毕业设计。但当前不能认定为 A：本地运行和前端构建未验证通过，部署不可访问，测试证据不足，论文/PPT 多处功能描述超过代码实现，若直接答辩容易在演示、测试真实性和实现细节追问中暴露问题。当前也不至于评为 C，因为核心代码、知识库、前端交互和论文材料均已具备，只要按优先级整改，仍可进入答辩准备。

1. 问题：后端依赖文件错误、环境无法运行。应补到哪里：代码/README。建议做法：修正 `backend/requirements.txt`，删除 `uvicorn main:app --reload`，补齐依赖后给出可成功启动的命令截图。

3. 问题：线上部署无法验证。应补到哪里：部署/README/PPT。建议做法：提供可访问的前后端完整部署，至少保证答辩现场能完成一次需求生成；若网络受限，准备本地演示方案。
4. 问题：前端硬编码本地后端地址。应补到哪里：代码/部署。建议做法：改为环境变量或相对路径 `/api`，与 Nginx/云部署代理配置一致。
5. 问题：论文中存在未实现或支撑不足的高级功能。应补到哪里：论文/PPT。建议做法：把多轮推理、规则引擎、知识图谱、批注修改、CSV/JSON 导出、时间轴等改为未来工作，或补齐真实实现。
6. 问题：测试证据不足。应补到哪里：论文/README/部署。建议做法：补充接口测试记录、典型输入输出 JSON、性能测试截图、验收表通过情况。
7. 问题：知识库数量和来源不一致。应补到哪里：论文/PPT/README。建议做法：统一为实际数据库统计值，或重新初始化后提供 `knowledge_module` 总数截图。
8. 问题：`micro_story` 相关叙述与代码矛盾。应补到哪里：论文/PPT。建议做法：删除相关字段和测试描述，或实现并展示该功能。

## 11. 简短结论

当前项目适合进入“答辩前集中整改”阶段，但不建议直接按现状定稿答辩。优先把本地运行、前端构建、部署演示和测试证据补齐，再同步收敛论文/PPT 中超出代码实现的表述；完成这些整改后，项目可以支撑一次较稳妥的毕业设计答辩。
