# 毕业设计整改任务清单

## 必须立即修改

1. 问题：后端依赖文件不可直接使用，`backend/requirements.txt` 第 6 行混入了 `uvicorn main:app --reload`。
   修改目标：保证后端可以按 README 安装依赖并启动。
   建议做法：删除该启动命令，只保留依赖包；补充版本号更好；重新验证 `pip install -r backend/requirements.txt` 和 `uvicorn main:app --reload`。
   影响位置：代码 / README / 部署。

2. 问题：当前检查环境无法导入 FastAPI，后端没有被实际启动验证。
   修改目标：提供一套可复现的后端运行环境。
   建议做法：创建干净虚拟环境，安装依赖，启动后访问 `/health`、`/docs`、`/preview_context`、`/generate_and_evaluate`，保存启动截图和接口返回截图。
   影响位置：代码 / 论文 / PPT / 部署。

3. 问题：前端 `npm run build` 失败，`vue-cli-service` 文件缺少执行权限。
   修改目标：保证前端能正常打包和部署。
   建议做法：重新安装前端依赖或修复可执行权限，验证 `npm run build` 成功；将构建成功结果写入 README 或部署说明。
   影响位置：代码 / 部署 / README。

4. 问题：云部署链接本次无法访问，线上演示能力无法判断。
   修改目标：答辩前必须有可打开、可演示主流程的系统入口。
   建议做法：重新检查 EdgeOne 部署状态、域名证书、访问 token 有效期；至少准备本地启动方案作为备选演示。
   影响位置：部署 / PPT / README。

5. 问题：前端接口地址硬编码为 `http://127.0.0.1:8000`，线上部署时容易请求答辩电脑本机端口。
   修改目标：让前端在本地和云端都能正确连接后端。
   建议做法：改为环境变量或相对路径 `/api`；部署时用 Nginx 或云平台代理转发到后端；README 中写清楚配置方法。
   影响位置：代码 / 部署 / README。

6. 问题：论文和 PPT 宣称多轮推理、规则引擎、知识图谱、时间轴、批注修改、局部重推理、CSV/JSON 导出等，但代码未看到实现。
   修改目标：论文、PPT 与实际系统保持一致。
   建议做法：未实现的内容改成“未来展望”或删除；只把关键词知识匹配、Prompt 约束、JSON 归一化、任务依赖图、Markdown 导出作为已实现功能写。
   影响位置：论文 / PPT。

7. 问题：论文中出现 `micro_story`、`generate_micro_story`，但当前代码没有该字段或函数，提示词还要求不要生成微型故事。
   修改目标：消除论文与代码的直接矛盾。
   建议做法：删除微型故事相关表述和测试项；如果确实要保留，就补齐后端字段、前端展示和测试截图。
   影响位置：代码 / 论文 / PPT。

8. 问题：论文验收测试表仍有“待填写”，性能测试和 100% 成功率缺少原始证据。
   修改目标：让测试章节能被答辩老师追问时站得住。
   建议做法：补真实测试记录，包括输入、接口、响应、耗时、是否通过、截图；验收表必须填“通过/不通过”和证明材料。
   影响位置：论文 / PPT / README。

## 建议尽快修改

1. 问题：知识库数量表述不一致，README 写 30+，PPT 写 42，本地数据库查询为 38。
   修改目标：统一知识库规模说明。
   建议做法：以实际数据库为准统一修改，或重新运行初始化脚本后给出数据库统计截图。
   影响位置：论文 / PPT / README。

2. 问题：论文/PPT 中写 Element Plus，但代码依赖是 Element UI。
   修改目标：技术栈表述准确。
   建议做法：统一改为 Vue2 + Element UI + ECharts，避免答辩时被指出技术栈不一致。
   影响位置：论文 / PPT / README。

3. 问题：README 的部署代理示例使用 `/api`，但前端代码没有按 `/api` 请求。
   修改目标：部署说明与实际代码一致。
   建议做法：要么修改代码走 `/api`，要么 README 写明前端需要配置后端完整地址；推荐改成 `/api` 代理。
   影响位置：代码 / README / 部署。

4. 问题：缺少单独的用户手册、部署验证记录和测试说明。
   修改目标：归档材料更完整，答辩前检查更容易通过。
   建议做法：新增简短文档，包含本地启动、API Key 配置、典型演示流程、常见失败处理、测试用例清单。
   影响位置：README / 文档 / 部署。

5. 问题：论文中很多截图位置需要确认是否清晰、是否与图题对应。
   修改目标：论文图表能真实支撑系统实现。
   建议做法：逐图检查图 4.1 到图 6.7，确保不是空白、占位图或与图题不符；优先放真实页面和接口返回截图。
   影响位置：论文。

6. 问题：PPT 偏概念说明，真实系统演示证据不足。
   修改目标：让答辩展示更像成果展示，而不是概念介绍。
   建议做法：增加一页完整演示路径和多张真实截图：输入需求、知识命中、用户故事、任务表、依赖图、代码生成、导出文件。
   影响位置：PPT。

7. 问题：缺少知识增强效果对比。
   修改目标：突出选题特色，避免被认为只是普通 LLM 包装页面。
   建议做法：选 2 到 3 个需求，展示命中知识库前后的差异，例如登录、支付、权限管理，说明补充了哪些前置条件、异常场景和安全约束。
   影响位置：论文 / PPT / 测试材料。

8. 问题：代码生成依赖外部模型，答辩现场可能受 API Key 或网络影响。
   修改目标：保证演示稳定。
   建议做法：准备固定样例输入、成功响应截图、导出的 Markdown 文件；必要时增加演示用缓存样例，但要在文档中说明用途。
   影响位置：代码 / 部署 / PPT。

## 可选优化

1. 问题：当前评价模块规则较简单，主要按字段存在性打分。
   修改目标：让“实验评估体系”更有说服力。
   建议做法：补充更细的质量规则，例如故事模板完整性、任务覆盖前后端数据库、依赖是否引用存在任务、知识命中项是否被体现。
   影响位置：代码 / 论文。

2. 问题：知识库匹配主要是关键词包含，论文中“术语消歧”支撑不足。
   修改目标：增强知识融入特色。
   建议做法：至少补充别名覆盖、命中模块展示、未命中提示和命中解释；暂时不要宣称向量匹配或复杂本体推理。
   影响位置：代码 / 论文 / PPT。

3. 问题：没有自动化测试。
   修改目标：降低答辩前回归风险。
   建议做法：增加后端最小接口测试和评价函数测试；前端至少保留构建验证记录。
   影响位置：代码 / 测试材料。

4. 问题：前端依赖图只有基本节点和连线。
   修改目标：提高展示效果。
   建议做法：增加点击节点后展示任务详情、所属故事和依赖信息；如果时间不够，只在 PPT 中准确描述现有能力。
   影响位置：代码 / PPT。

5. 问题：README 内容较全，但缺少“一分钟演示脚本”。
   修改目标：方便答辩前快速复现。
   建议做法：写明演示输入、点击顺序、预期看到的结果、可能失败原因和备用截图位置。
   影响位置：README / PPT。

6. 问题：数据库和 `data/knowledge.json` 同时存在，容易让检查者困惑。
   修改目标：明确当前系统实际使用的数据源。
   建议做法：在 README 和论文中说明运行时以 SQLite 为准，JSON 是早期样例或初始化参考；不使用的材料可放入说明而不是混淆为主数据源。
   影响位置：README / 论文。
